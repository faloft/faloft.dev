# Справочник школ из 50 280 строк: ручная чистка → детерминированный скрипт

*ROUND, 2025. Технический кейс: формализация правил + 8-шаговый скрипт на pandas+regex.*

## Проблема

Заказчик ROUND — продукт с регистрацией, где пользователь выбирает школу из справочника. Источник — выгрузка из Яндекс.Карт по категории «Образование», **50 280 строк**: общеобразовательные школы вперемешку с детсадами, кружками, библиотеками, автошколами, церковными учреждениями, корпусами одной школы как отдельные записи.

До меня справочник чистился руками — спорные случаи по чувству. На 50к строк это невоспроизводимо: «по чувству» расходится между людьми и между днями.

## Что сделал

- **Формализовал правила** до написания кода. Целевые: СОШ, гимназии, лицеи, интернаты, детские дома, частные школы. Нецелевые: детсады, ВУЗы, колледжи, кружки, корпуса при наличии базовой записи, всё религиозное, медицинское, исправительное, дополнительное.
- **Скрипт `clean_v10.py` на pandas + regex, 8 шагов:** нормализация ОПФ → ключевые корни → корпуса → нецелевые категории → гео-мусор → дубли по «название+адрес» → дубли по адресу → blacklist.
- **18 итераций с РП через файл удалённых с причинами.** Каждый цикл: новая версия → файл удалённых с колонкой «Причина» → РП фильтрует, проверяет 5–10 строк → говорит, что добавить или убрать → следующая версия. Без этого файла верификации нет — и доверия нет.

## Результат

**50 280 → 41 980 строк.** ~8 300 нецелевых вычищено, версия v18 принята РП. Каждый прогон — три артефакта: чистый файл, удалённые с причинами, отчёт по шагам. Любая правка откатывается и проверяется.

## Почему это работает

Дедуп по «Название + Полный адрес», не по названию — иначе теряются разные «Бобровские школы» в разных районах. Корпуса — отдельный механизм: есть базовая запись на адресе → удаляем корпусную, нет → оставляем и обрезаем суффикс. Корни вместо точных слов сокращают список ключевых слов в разы.

## Принцип

**Сначала формализовать правила, потом писать код. ИИ — соавтор правил, а не чистильщик.**

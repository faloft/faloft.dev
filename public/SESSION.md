## Git-стратегия для v1 (зафиксировано 2026-04-27)

**Локальный git** и **GitHub-push** — два разных шага, не путать:

1. **Локальный `git init`** — делается рано, как только в проекте есть что коммитить. Ничего не уходит наружу. Даёт защитную сетку: можно откатить любую правку, любой эксперимент, любой сломанный рефакторинг. Без этого правки необратимы.
2. **GitHub-push** — отдельная операция, делается **только когда готовы к первому деплою на Vercel**. До push — упоминание GitHub в футере не имеет смысла (ссылка ведёт в никуда).

**План для текущей итерации лендинга:**

- Локальный git заводим **до начала следующего крупного шага** (полные страницы кейсов через Content Collections) — чтобы дальше работать с откатной сеткой.
- GitHub-репо создаём + push **после** того, как лендинг готов к первому показу: hero, что даю, принцип, кейсы (тизеры + полные), контакты, футер.
- После первого push → подключаем Vercel → получаем https-URL → **только тогда** добавляем «→ Исходник» ссылкой в футер.

**Что не делать:**

- Не пушить в GitHub раньше первого деплоя — ссылка в футере на пустой/недоделанный репо смотрится плохо.
- Не оставлять проект без локального git дольше, чем на 1–2 правки. Это лотерея на необратимость.

**GitHub-аккаунт пользователя:** `https://github.com/faloft` — туда создаём репозиторий и пушим перед Vercel-деплоем. Имя репо — `portfolio` или `landing` (решим перед push).

**Локальный git инициализирован 2026-04-27.** Первый коммит `08a13f5` — лендинг v1 (scaffold + все секции). Дальше: коммитить после каждого крупного шага.
